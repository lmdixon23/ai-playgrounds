↓ Read the explanation

"Find the shortest path from A to B" is one of the most useful things a computer can do. Your phone's maps app does it thousands of times a day. Every video game with NPCs that walk around uses it. The four algorithms in this visualization are the classical building blocks that every modern routing algorithm descends from.

Breadth-First vs Depth-First

BFS explores cells in order of how many steps they are from the start: first all 1-step neighbors, then all 2-step neighbors, then all 3-step neighbors, and so on. When it reaches the goal, it has automatically found the shortest path because it never considers a longer path before a shorter one. The price is that it visits a lot of squares - essentially a full circle expanding outward from the start.

DFS goes the opposite direction: pick a neighbor, recurse, pick that neighbor's neighbor, recurse. It dives deep before exploring breadth. DFS is wonderful for problems where any path will do (like exploring a maze you've never seen) but it is famously bad at finding short ones: it'll happily wander to the bottom-left corner of the grid before noticing the goal is two squares to the right.

A* and Why It's Different

BFS visits many cells because it has no idea where the goal is. A* fixes this with a simple trick: when deciding which cell to explore next, it picks the one that minimizes f(cell) = g(cell) + h(cell), where g is the actual cost from the start and h is an estimate of remaining cost to the goal. On a grid, a good estimate is the Manhattan distance (number of squares you'd need to walk if there were no walls).

BFS asks "which cell is closest to the start?" A* asks "which cell is on the best complete path I've imagined so far?" That tiny shift in question makes the search aim at the goal instead of expanding blindly in all directions.

Dijkstra's algorithm is what you get if you remove the h term entirely (h = 0 everywhere). On a uniform grid like this one, Dijkstra and BFS behave identically. Dijkstra only earns its keep when edges have different weights.

When Each One Wins

Draw a long, open corridor from the start to the goal with no obstacles. Run BFS, then A*. A* will trace almost a straight line to the goal; BFS will fan out in a wide arc and visit five to ten times as many cells. Now build a maze with walls funnel-shaped toward the start : the heuristic is misleading, and A* expands roughly as many cells as BFS. The lesson is that A*'s speed advantage is exactly proportional to how informative the heuristic is.

What This Simplifies

Every cell here has cost 1. Real pathfinding (Google Maps, game AI, robotics) uses weighted graphs where different roads cost different amounts. The algorithms here generalize directly: BFS becomes Dijkstra on weighted graphs; A* keeps the same structure but with a more sophisticated heuristic. The version of A* used in production game AI typically uses "jump-point search" (a navigation-mesh shortcut) and "any-angle" path smoothing: both built on the A* skeleton you see here.

For teachers

Curriculum: Problem Representation & Search. Distinguish uninformed vs. informed search; predict node-expansion order; understand how a heuristic narrows the frontier.

Pre-exploration prompts:

  • If both BFS and A* find the shortest path, why prefer one over the other?
  • What's a "frontier"? What's a "visited" set? Why both?
  • What information does A* have that BFS doesn't?

Post-exploration prompts:

  • Draw a wall configuration where DFS finds a longer path than BFS. Why does that happen?
  • Move the goal far from the start in one direction. Compare A*'s visited count to BFS's. Why are they so different?
  • Build a maze where A* expands almost as many nodes as BFS. What property of the maze causes that?

Common misconceptions defeated: "A* is always better than BFS" (not when the heuristic is uninformative); "DFS is faster because it goes deep first" (often finds bad paths and has to backtrack); "Dijkstra needs weighted edges to be interesting" (equals BFS on uniform grids: that's the lesson).
