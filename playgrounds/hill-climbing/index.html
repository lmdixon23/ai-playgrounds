↓ Read the explanation

Local search algorithms don't "see" the whole problem. They see only the current state and a handful of "neighbor" states reachable in one move. They have to decide where to go next from that tiny window. This applet pits six of the classic strategies against three classic problems so you can watch which strategies get stuck and which escape.

The two-axis view of local search

Every local search algorithm can be described by two choices: which neighbor to consider and which to accept. Simple hill-climbing considers neighbors one at a time and accepts the first improving one. Steepest-ascent considers all neighbors and accepts the best. Stochastic HC weights by improvement. First-choice HC samples randomly. Random-restart adds a "when stuck, start over" outer loop. Simulated annealing adds the radical option: sometimes accept a worse move.

Why greedy methods get stuck

On a multi-modal problem (N-queens, TSP, the function landscape with five minima), greedy methods reach a local optimum and stop. From that local optimum, every neighbor looks worse — so a greedy algorithm has nowhere to go, even though a globally better state exists a few steps away on the other side of a hill. This is the central pathology that local search has to solve.

Watch steepest-ascent on N-queens. Most runs end at a state with 1 or 2 remaining attacks — not zero. The greedy algorithm has done everything it can without making a move that momentarily looks bad. Sometimes the right move is a worse-looking one.

The two ways out of local optima

Random-restart is the brute-force fix: when stuck, throw the state away and start fresh from a random configuration. Run it long enough and you'll eventually start near the global optimum. Watch the cost chart "jump up" when the restart happens — that's the cost of starting over.

Simulated annealing is the elegant fix: accept worse moves with probability exp(−ΔE/T), where T is a temperature that decreases over time. Early on (high T), almost any move is accepted, so the algorithm wanders freely. Later (low T), only improving moves are accepted, so it behaves like greedy hill-climbing. The trick is the cooling schedule: cool too fast and you freeze in a bad spot; cool too slow and you waste time wandering.

Why each problem teaches something different

  • TSP uses 2-opt neighbors (reverse a tour segment). The neighborhood is huge — 66 neighbors per state with 12 cities — so steepest-ascent is expensive but effective.
  • N-queens has a tiny neighborhood (move one queen in its column) but a rugged cost surface. Local optima are common.
  • Function landscape is the canonical Bret-Victor-style picture: you can literally see the local minima as dark spots on the heatmap, and the trajectory yellow line shows the algorithm's path. Try simulated annealing here to see the wandering phase.

What this simplifies

Real local search uses tabu lists (forbidden recent states), problem-specific moves, population-based methods (genetic algorithms, particle swarm), and parallel restarts. Real SA cooling schedules are designed with theoretical guarantees ("logarithmic cooling is guaranteed to find the global optimum given infinite time"). But the heart — propose a neighbor, decide whether to accept — is exactly what runs here.

For teachers

Curriculum: Local Search. Russell & Norvig Chapter 4. Distinguish neighbor structure from algorithm choice; experience why local optima trap greedy algorithms; understand temperature in simulated annealing.

Pre-exploration prompts:

  • If you're hiking blind in fog and want the highest peak, what strategy do you use?
  • What's a "local optimum"? Can a greedy algorithm get stuck in one?
  • Why might it sometimes be WORTH taking a step that makes things worse?

Post-exploration prompts:

  • Run steepest-ascent on TSP a few times. Why does the cost curve flatten before reaching zero?
  • Now run random-restart on the same TSP. Watch the cost "jump up and back down" each restart. Why is this an improvement?
  • Run SA. Watch accept/reject early vs late. Why are early bad-moves useful, and why does SA stop accepting them later?
  • Set cooling to 0.90 (fast). What does SA degenerate into when T → 0 immediately?
  • Try N-queens with steepest-ascent. Why does the algorithm often stop at 1-2 attacks?

Common misconceptions defeated: "more random = better" (pure random search has no signal); "SA is always better than HC" (no — for unimodal problems HC is faster); "the algorithm sees the landscape" (no — only the current state and its neighbors).
