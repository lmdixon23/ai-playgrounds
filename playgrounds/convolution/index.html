↓ Read the explanation

A convolutional neural network (CNN) sees images by repeatedly applying a small grid of numbers : a kernel: to local neighborhoods of pixels. This applet lets you draw on a 32×32 canvas, pick a 3×3 kernel, and watch what falls out the other side. The mechanism is far simpler than the name suggests.

What "convolution" actually means here

For every pixel in the output, look at the 3×3 neighborhood centered on that pixel in the input. Multiply each input pixel by the corresponding cell of the kernel. Sum the nine products. That sum becomes the output pixel. Slide the 3×3 window one pixel over and repeat, until every output pixel is computed. That's the entire operation.

The Sobel-X kernel [[−1,0,1],[−2,0,2],[−1,0,1]] measures the horizontal gradient. Wherever the input transitions from dark to light in the horizontal direction, the convolution output spikes positive. Where it transitions light to dark, it spikes negative. The "edges" you see in the output are literally these spikes, brightened.

The detection of vertical edges is not pattern recognition. It is nine multiplications and an addition, repeated for every output pixel. The kernel just happens to be designed so that the arithmetic gives you a large result exactly when an edge exists.

Why the absolute value matters

Sobel-X gives positive numbers for one direction of edge and negative for the other. If you display the raw output, the "going light-to-dark" edges look black and the "going dark-to-light" edges look bright. Most visualizations take the absolute value to show edge strength rather than edge direction. Toggle "Take absolute value" to see the difference.

What ReLU does

In a real CNN, every convolutional layer is followed by a non-linear activation: usually ReLU, which clips negative values to zero. Toggle "Apply ReLU" with the Sobel-X kernel: half the edge information disappears. That's the cost CNNs pay for the flexibility ReLU provides; they compensate by learning both a kernel and its mirror, so no information is lost across the layer.

Stacking layers: pooling and conv-on-conv

A single 3×3 convolution sees only 9 pixels at a time. To recognize larger structures (a whole letter, a face, an object), CNNs stack convolutions and interleave them with downsampling operations. The layer-mode dropdown above demonstrates the two basic stacking moves:

  • Conv + max-pool 2×2. After the convolution, take 2×2 blocks of pixels and keep only the brightest from each block. The output is half the resolution (16×16 instead of 32×32). This (a) reduces computation for the next layer, and (b) gives the network a small dose of translation-invariance: shifting the input by one pixel doesn't change the output much.
  • Conv → conv (stacked). Two 3×3 convolutions in a row. The first sees 3×3 = 9 input pixels per output pixel; the second sees 3×3 = 9 first-layer outputs per its own output pixel. Each of those 9 first-layer outputs was itself computed from 3×3 input pixels. So each pixel of the final output is a function of a 5×5 region of the original input. Stacking n 3×3 convs gives a receptive field of 2n+1 in each direction: without ever using a kernel larger than 3×3.
Modern CNNs almost never use kernels larger than 3×3. They get arbitrary receptive fields by stacking. Two stacked 3×3 convs have the same receptive field as one 5×5 conv but use fewer parameters (18 vs 25) and produce a more expressive function (because of the non-linearity between them). This is the central insight of VGGNet (2014) and every CNN since.

How CNNs go from one kernel to thousands

A real CNN doesn't use one kernel: it uses dozens at the first layer, hundreds in the middle, thousands at the end. And the kernels aren't hand-designed (like our edge detectors): they're learned from data by gradient descent. The training process discovers kernels that are useful for the task: early-layer kernels typically end up looking like edge and color detectors; middle-layer kernels detect textures and shapes; late-layer kernels detect entire objects (eyes, wheels, faces).

What this simplifies

Real CNNs use 3D kernels (height × width × channels), apply many kernels per layer (forming a "feature map" stack), interleave convolutions with pooling and batch normalization, and use backprop through the convolution operation to learn the weights. But the basic operation - multiply, sum, slide, repeat: is exactly what runs above.

For teachers

Curriculum: Vision with CNNs. Convolution as multiply-and-sum; hand-designed vs learned filters; the role of activation functions after convolution. The 🎯 Learn the filter mode is the bridge from the neural-network lesson: a CNN learns its filters by gradient descent the same way last week's neural network learned its weights. The Inspect & scan panel lets students check one convolution by hand (the sum stays hidden until they click); the Learner-vs-target view shows a single image can't fully pin down a filter: different kernels can give the same output (under-determination).

Pre-exploration prompts:

  • How might a computer tell that a picture contains "an edge"?
  • If I gave you a 3×3 number grid, what could you do with it to highlight an edge in a picture?
  • Why do CNNs use LOTS of small filters instead of one big one?

Post-exploration prompts:

  • Draw a vertical line. Apply Sobel-X. Apply Sobel-Y. Which one lights up? Why?
  • Apply the Identity kernel. Why does the output look like the input?
  • Apply Sobel-X with "absolute value" OFF. The right side of the edge is black. What's mathematically happening?
  • Build a custom kernel that highlights just diagonal edges. (Hint: rotate the Sobel-X pattern.)
  • Apply Blur, then Edge. Then try Edge, then Blur. Same result? Why does order matter?

Common misconceptions defeated: "CNNs invented edge detection" (no: classical CV used hand-designed filters long before; CNNs LEARN them); "ReLU is the model" (no: it's just a clip; the convolution does the work); "more filters = always better" (more filters add capacity but also computation and overfitting risk).
